A.I. Brief

How it works 

When a user taps "AI Brief" on a matched player in Discover, the PlayerBrief component immediately sets a loading state and shows "Generating…" — that's the required loading state, up front. It then POSTs to /.netlify/functions/ai-brief with { mode: "matchup", player: {name, ntrp, format}, session }. The function takes that payload, builds a 3-bullet match-prep prompt, and forwards it to OpenAI's /v1/chat/completions endpoint using gpt-4o-mini at temperature 0.7. When OpenAI answers, the function extracts choices[0].message.content, returns 200 { brief }, and the client renders a dismissable "AI Match Prep" card. If OpenAI is unreachable or returns a bad status, the function sends back a 502/500 with an error body, the client's catch block runs, and the user sees an actual error message, no silent failure, which is another explicit rubric item. The same function does double duty: mode: "matchup" produces the brief, while mode: "chat" powers the AI Coach tab by sending the last 6 conversation turns for a coaching reply. One deployed function, two features. 

Why it was designed this way The serverless proxy exists because browser code is public — anyone can open DevTools and read every request the client makes, so calling OpenAI directly from React would expose the API key to the world. Routing through the Netlify function keeps OPENAI_API_KEY in a Netlify environment variable, off the client and out of the repo entirely, which satisfies the hard rule about credentials. It also hits the "genuinely useful" bar directly: the app takes real data (this specific opponent's NTRP, the format, the session), constructs a prompt from it, and returns decision support for a real task, literally the example the guidelines give of what qualifies, as opposed to a bare chatbot wrapper. And gpt-4o-mini is the sensible model choice: cheap and fast for short structured output like three bullets.
