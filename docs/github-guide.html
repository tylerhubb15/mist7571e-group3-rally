What is GitHub?

🏠 The Metaphor — Google Docs for Code Imagine six people writing a group essay in Google Docs. Everyone can see the latest version, work on their own section, and save a history of every change ever made. GitHub is exactly that — but for code. Instead of "the document," you have "the repository (repo)." Instead of typing in Google Docs, you write code in VS Code and then "upload" your changes to GitHub.

GitHub does three things for us:

📦
Stores the Code
One central place online where the entire Rally project lives. Anyone on the team can access it.
🕐
Tracks History
Every change ever saved is recorded with who made it and when. You can always roll back if something breaks.
🔀
Manages Teamwork
Six people can work simultaneously without overwriting each other's work — using branches.

Git vs. GitHub — these are two different things that work together:

  • Git — a program installed on your computer that tracks changes to files.
  • GitHub — a website that hosts your Git repository online so the whole team can access it.
🎾 Tennis Metaphor Git is your personal scorebook — you write down every point you win on your court. GitHub is the tournament leaderboard — you post your scores there so everyone can see the official standings.

What is a Branch, really?

🌳 Metaphor — The Tree Imagine the project is a tree. The trunk is main — the official, working version of the app. Each team member grows their own branch off the trunk. You can add leaves (code) to your branch without touching the trunk. When your branch is ready, you merge it back into the trunk. The trunk stays clean and working at all times.
main ●────────────────────────────────────────● (the trunk — always working) │ ↑ merge PR │ │ role2 ────────●──●──●──●──●──●──●──●──●──● (your branch — AI integration) role4 ─────────────●──●──●────────────────── (profile developer's branch) Each person works on their own branch simultaneously. Getting Started — One-Time Setup

Do these steps once on your computer. After that, you just use the daily workflow.

Step 1 — Install the tools

  • A
    Install Git — Download from git-scm.com and run the installer. Accept all defaults.
  • B
    Install Node.js — Download the LTS version from nodejs.org. This also installs npm.
  • C
    Install VS Code — Download from code.visualstudio.com if you don't have it.
  • D
    Create a GitHub account — Go to github.com/join. Use your UGA email. Then tell Tyler your username so he can add you to the repo.

Step 2 — Tell Git who you are

Open VS Code, press Ctrl + ` (backtick) to open the terminal, then type:

git config --global user.name "Your Full Name"
git config --global user.email "yourname@uga.edu"

You only do this once. It labels all your commits with your name.

Step 3 — Clone the repository

Cloning means "download the entire project to my computer."

# In your terminal, navigate to where you want the project folder:
cd Documents          # or wherever you want it

# Clone the repo — this creates a new folder called mist7571e-group3-rally:
git clone https://github.com/tylerhubb15/mist7571e-group3-rally.git

# Move into the project folder:
cd mist7571e-group3-rally
💡 Easier way in VS Code Press Ctrl+Shift+P, type "Git: Clone", paste the repo URL, and choose where to save it. VS Code handles everything else.

Step 4 — Switch to your branch

After cloning you are on main. Switch to your assigned branch before touching anything:

# Switch to your branch. Replace role3 with your actual role number:
git checkout feature/role3-sessions-view

# Confirm you're on the right branch (you'll see a * next to it):
git branch
  feature/role2-ai-integration
* feature/role3-sessions-view    ← you are here
  main

Step 5 — Install dependencies and run the app

# Install all packages the project needs:
npm install

# Start the local development server:
npm run dev

  VITE v5.x.x  ready in 312 ms
  ➜  Local:   http://localhost:5173/
  ➜  Network: http://192.168.x.x:5173/

Open http://localhost:5173 in your browser — you should see the Rally app running. The app works fully on fake/mock data, so no API keys are needed just to run it.

✅ You're all set! If the app loads in the browser, your environment is working. Stop the server with Ctrl+C in the terminal.

Daily Workflow — Every Time You Work

📰 Metaphor — The Morning Newspaper Before you start writing today's story, you pick up the morning paper to see what happened since yesterday (pull). You write your story on your own notepad (edit files on your branch). When you're happy with it, you save a draft with a headline (commit). Then you submit it to the editor (push). At the end of the week, the editor reviews it and prints it (pull request).

The 5-step cycle — repeat every time you work

  • 1
    Pull first — always download the latest changes before you start working. 
    git pull origin feature/role3-sessions-view
# (replace with your branch name)
    ⚠️ Don't skip this If you forget to pull and a teammate has pushed changes, your code will get out of sync. Always pull first.
  • 2
    Make sure you're on your branch — before writing any code. 
    git branch         # See which branch you're on
git checkout feature/role3-sessions-view   # Switch if needed
  • 3
    Write your code — edit files in VS Code as normal. The app reloads live at http://localhost:5173 as you save.
  • 4
    Stage and commit — save a snapshot of what you did. 
    # See what files you changed:
git status

# Stage all your changes (the . means "all files"):
git add .

# Save a commit with a clear message describing WHAT you did:
git commit -m "add session card component with pending/confirmed states"
    ✅ Good commit messages Write short, specific descriptions: "add loading spinner to AI brief", not "updated stuff". Future-you will thank present-you.
  • 5
    Push — upload your commits to GitHub. 
    git push origin feature/role3-sessions-view
    Done. Your work is now saved on GitHub. Anyone on the team can see your branch.

Commit and push as often as you like — even if your work is unfinished. Frequent small commits are better than one big one at the end.

Pull Requests — Merging Your Work

When your feature is complete and tested, you open a Pull Request (PR) to merge your branch into main.

✍️ Metaphor — Submitting a Paper for Grading You've been drafting your section of the group project on your own laptop (your branch). When it's ready, you submit it to the group (open a PR). A teammate reviews it, suggests edits if needed, and then approves it. Once approved, your section gets added to the shared final document (merged into main).

How to open a Pull Request

  • 1
    Push your branch to GitHub one final time: git push origin feature/your-branch
  • 2
    Go to github.com/tylerhubb15/mist7571e-group3-rally
  • 3
    GitHub will show a yellow banner: "Your branch had recent pushes — Compare & pull request." Click it.
  • 4
    Write a title and description explaining what you built. Then click "Create pull request."
  • 5
    Tag a teammate (Role 5 — QA or Tyler) as a reviewer.
  • 6
    Once approved, click "Merge pull request". Your code is now in main.
💡 Tip You don't have to wait until everything is done to open a PR. You can open a Draft PR early — it signals "I'm working on this" without merging. Just click the arrow next to "Create pull request" and choose "Create draft pull request."

What happens during review

The reviewer (Role 5) will look at your code and may:

  • ✅ Approve — your PR can be merged
  • 💬 Request changes — they'll leave comments on specific lines. Fix them, commit, push, and the PR updates automatically.

Using VS Code — No Terminal Needed

VS Code has a built-in Git panel that lets you do most things without typing commands. Here's how to use it.

The Source Control panel

Click the branch icon in the left sidebar (or press Ctrl+Shift+G).

Source Control — VS Code
SOURCE CONTROL ↺ ↓ ↑ ✓ ···

MESSAGE
add session card with status states ← type your commit message here

✓ Commit & Push ← click this button to commit + push at once

CHANGES (3)
M src/components/CalendarView.jsx M = modified
A src/components/SessionCard.jsx A = new file added
M src/App.jsx

Click the + icon next to a file to stage it, or click + next to CHANGES to stage all

Committing via VS Code (click-by-click)

  • 1
    Open the Source Control panel (Ctrl+Shift+G)
  • 2
    You'll see all your changed files listed under CHANGES
  • 3
    Click the + next to "CHANGES" to stage all files
  • 4
    Type your commit message in the Message box at the top
  • 5
    Click the dropdown arrow on the blue Commit button and choose "Commit & Push"

Pulling via VS Code

  • 1
    Open Source Control panel
  • 2
    Click the ··· (three dots) menu at the top
  • 3
    Choose Pull

Switching branches in VS Code

Look at the bottom-left corner of VS Code — it shows your current branch name (e.g., feature/role3-sessions-view). Click it to switch branches without using the terminal.

✅ Recommended Extensions Install these from the VS Code Extensions panel (Ctrl+Shift+X):
  • GitLens — shows who wrote each line and the full commit history inline
  • GitHub Pull Requests — create and review PRs without leaving VS Code
  • ESLint — catches code quality issues automatically

Handling a merge conflict

A conflict happens when two people edit the same line of the same file. VS Code makes this easy to resolve:

Merge Conflict in VS Code
Accept Current Change | Accept Incoming Change | Accept Both | Compare Changes

<<<<<<< HEAD (Current Change)
const title = "Sessions"; ← your version
=======
const title = "Your Sessions"; ← teammate's version
>>>>>>> feature/role3 (Incoming Change)

Click "Accept Current Change" or "Accept Incoming Change" (whichever is correct), then save, commit, and push. Conflict resolved.

🎾 Conflict Metaphor Two players both tried to book Court 3 at the same time. A conflict is GitHub saying "I can't figure out whose booking wins — you two decide." You look at both requests and pick one (or combine them), then re-submit.

Team Rules & Common Mistakes

✅
Always Do
• Pull before you start working
• Work on your own branch only
• Commit with clear messages
• Push at the end of every session
• Open a PR when a feature is done
❌
Never Do
• Push directly to main
• Commit a .env file
• Commit node_modules/
• Put API keys in any source file
• Force-push (--force)

The .env file — CRITICAL security rule

🚨 Never commit .env to GitHub The .env file holds secret API keys (like the OpenAI key). If it ends up on GitHub — even in a private repo — you risk having the key stolen and your account charged thousands of dollars in AI API fees. The .gitignore file already blocks it, but always double-check with git status before pushing. If you ever see .env in the changes list — stop and ask Tyler before doing anything.

Use .env.example instead — it's an empty template showing what keys are needed, with no real values. That file IS safe to commit.

# .env.example (safe to commit — shows structure only)
VITE_SUPABASE_URL=your_supabase_url_here
VITE_SUPABASE_ANON_KEY=your_anon_key_here
OPENAI_API_KEY=your_openai_key_here

# .env (NEVER commit — holds real secrets)
VITE_SUPABASE_URL=https://xyzcompany.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1...
OPENAI_API_KEY=sk-proj-...
