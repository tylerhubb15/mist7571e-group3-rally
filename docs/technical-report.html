The Big Picture

Rally is a web app that helps recreational tennis players find compatible hitting partners in Athens, GA. Think of it as a dating app for tennis — you fill in your skill level, schedule, and what you're looking for, and the app ranks nearby players by how well they match you.

🎾 The Restaurant Metaphor Imagine the whole app is a restaurant. React is the dining room — what the customer sees and interacts with. Vite is the kitchen equipment that cooks everything fast. Netlify is the building itself — the address people go to. GitHub is the head chef's recipe book — every change is recorded and any chef can work from it. Supabase (Phase 2) is the walk-in fridge — persistent storage. And OpenAI is the sommelier — it reads your table and suggests the perfect pairing.

The app currently runs on mock/fake data (Phase 1). In Phase 2, it connects to a real Supabase database. The AI feature (pre-session brief) is the bridge between the two phases — it runs server-side today but the real OpenAI call is still to be implemented.

How React Works in This App

🧩 The Lego Metaphor React breaks the UI into reusable pieces called components — like Lego bricks. Each brick knows how to draw itself. You snap bricks together to build the full app. When a piece of data changes (e.g., the user changes their NTRP rating), React automatically redraws only the bricks that use that data — no manual DOM manipulation needed.

The two rules React runs on

📊
State — "memory"
Variables that React watches. When state changes, React re-renders. Example: const [tab, setTab] = useState("discover") — changes which tab is visible.
📬
Props — "messages"
Data passed from a parent component to a child. Example: <Profile me={me} setMe={setMe} /> — the profile receives the user object and a setter.
⚙️
Hooks — "superpowers"
useState gives state. useMemo caches expensive calculations (like re-ranking 7 players). useEffect runs side effects (API calls, timers).

How state flows top-down in Rally

All the important state lives in App.jsx at the top. It passes pieces of that state down to child components via props. When a child needs to change something, it calls a function that was passed down from App.

// App.jsx — the "brain" of the app
const [tab, setTab]         = useState("discover");   // which screen is visible
const [me, setMe]           = useState(ME);            // the logged-in user's profile
const [sessions, setSessions] = useState(SEED_SESSIONS); // all booked sessions
const [modal, setModal]     = useState(null);          // which modal is open

// State flows DOWN via props:
<Discover me={me} onPropose={(p) => setModal(p)} />
<Profile  me={me} setMe={setMe} />
<CalendarView sessions={sessions} setSessions={setSessions} />
💡 Why this matters for the rubric The rubric specifically requires "props and state used correctly." The current structure is clean — state lives at the right level and flows predictably. Don't move state into child components unless it's local-only.

The Matching Engine

The heart of the app. Located in src/lib/matching.js, the scoreOf(me, player) function takes two player profiles and returns a compatibility score from 0–100.

🎯 The Compatibility Metaphor Think of it like a job interview scoring sheet. You score a candidate on four criteria and weight each one. The candidate with the highest total weighted score gets the offer. Rally does the same: it scores every player on four dimensions and ranks them by total score.

The formula (weighted scoring)

DimensionWeightHow it's calculated
Skill (NTRP)35%Closer NTRP ratings = higher score. Max 1 point if exact match; 0 if >1.5 levels apart.
Availability30%Shared time slots ÷ min(my slots, their slots). More overlap = higher score.
Distance20%Linear scale: 0 distance = 1.0; beyond search radius = 0.
Intent15%Same intent = 1.0; one step apart = 0.6; two+ steps = 0.25. (Casual → Drilling → Competitive → Match)
 
// src/lib/matching.js — the exact formula
export function scoreOf(me, player) {
  const skill  = Math.max(0, 1 - Math.abs(me.ntrp - player.ntrp) / 1.5);
  const dist   = Math.max(0, 1 - player.dist / me.radius);
  const overlap = player.slots.filter((s) => me.slots.includes(s));
  const avail  = me.slots.length ? overlap.length / Math.min(me.slots.length, player.slots.length) : 0;
  const gap    = Math.abs(intentRank(me.intent) - intentRank(player.intent));
  const intent = gap === 0 ? 1 : gap === 1 ? 0.6 : 0.25;

  const total = skill * 0.35 + avail * 0.3 + dist * 0.2 + intent * 0.15;
  return { total: Math.round(total * 100), parts: { Skill, Avail, Dist, Intent }, overlap };
}
✅ This formula is the same one in supabase/schema.sql The Phase 2 Postgres function find_matches() implements identical weights so server-side ranking matches client-side ranking.

The AI Integration — What's Missing

This is Role 2's primary deliverable and the highest-weighted single component on the rubric (20 pts). Here's how it's designed to work and what still needs to be built.

🤝 The Personal Trainer Metaphor When you book a training session with a new gym partner, the gym's AI assistant reads both your profiles and writes a one-page brief: "You're evenly matched on skill, but Alex prefers baseline rallies while you like net approaches — here's a warm-up drill that suits both styles." That's the pre-session brief. It reads structured data and returns personalized advice. That's exactly what OpenAI needs to do here.

✅
Server-side isolation
The architecture is correct. The API key will live in Netlify's environment variables dashboard — never in the code or repo.

What still needs to be built

❌

Replace the stub with a real fetch to api.openai.com/v1/chat/completions. Build a structured prompt from the two player profiles.

A React component in CalendarView that calls the function and shows the result with a loading state and error state.
❌
OPENAI_API_KEY in Netlify
Set in Netlify dashboard → Site settings → Environment variables. Never in .env, never in any committed file.

Phase 2 — The Supabase Backend

Phase 2 replaces mock data with a real database. The scaffolding is already written in src/lib/services.js and src/hooks/hooks.jsx. The supabase/schema.sql file defines the Postgres tables. Nothing needs to be redesigned — it just needs to be switched on.

🔑
Auth
Supabase handles login via Google/Apple/GitHub OAuth. useAuth() hook in hooks.jsx is ready.
🗃️
Database
Tables: profiles, availability_slots, sessions, messages. The find_matches() Postgres function runs the same scoring logic as the client.
⚡
Realtime
Supabase Realtime will power live message updates in the Messages tab via WebSocket subscriptions.
💡 Phase 2 is not required for the rubric The project guidelines say localStorage is acceptable for Phase 1. Supabase is the extension that pushes into A-range. Profile data persistence (Role 4) should use localStorage first, then optionally Supabase.
